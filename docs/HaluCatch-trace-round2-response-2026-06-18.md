## 修复 1：README 新增「常见问题」章节

**位置**: `README.md`，「许可」章节之前追加

```markdown
## 常见问题

**Q: HaluCatch 需要联网吗？**
A: 不需要。全程离线运行，仅扫描本地文件夹中的 SKILL.md 和 .py 文件。不会发起任何网络请求。

**Q: 我的 Skill 没有 .py 文件，能用吗？**
A: 可以。HaluCatch 会自动分类为「纯方法论型」并跳过地基/代码检查，只评估方法论和护栏。

**Q: 审查结果说「护栏薄弱」，怎么修？**
A: 看同目录下的 `-行动版.md` 报告，它包含每条风险的修复方案和 feedback.md 模板，按清单逐项改即可。

**Q: 为什么工具库型 Skill 的护栏分数看起来比分析型低？**
A: 护栏检查按类型分层——工具库型只查 5 项核心项（跳过不必要的数据来源/时效性/置信度检查），分母不同，分数不可直接比较。

**Q: 能一次审查多个 Skill 吗？**
A: 当前版本暂不支持批量模式。你可以逐个运行，批量功能已在 roadmap 中。

**Q: 出现「网络问题卡住」是 HaluCatch 的问题吗？**
A: 不是。HaluCatch 是纯本地工具，不走网络。如果执行卡住，大概率是 AI 对话环境超时或目标路径过大导致扫描耗时。
```

## 修复 2：报告 info 级输出通俗化过滤

**位置**: `halucatch_core.py` → `generate_report()`

**问题**: 报告中的 `info` 级输出偶尔包含原始字段名（如 `min_py_lines`、`total -= 1` 的逻辑痕迹），非技术用户看到会困惑。

**修复**: 在通俗版报告中自动过滤所有 `info` 级别的输出项（不在通俗版展示），同时在 `context_map` 中补三条映射：

```python
context_map = {
    # ... 现有映射 ...
    '未声明数据来源限制': '没说明数据从哪里来、覆盖什么范围，不同 AI 可能用不同数据源',
    '未要求置信度声明': '没要求 AI 标注对结论有多大把握，容易把不确定的事说得很肯定',
    '引用.*不存在的文件': '说明书写了要用某个文件，但文件夹里没有——大概率是文件名写错了或忘了放',
}
```

**改动量**: 3 行 context_map 扩展，零逻辑变更。
